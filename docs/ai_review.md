# 外部AIレビューの回し方

複数のAIモデルにこのサイトを見てもらうときの手順。**発見層に置き、判断層に置かない**。

## 線引き

| 使う | 使わない |
|---|---|
| UI・導線・体験の問題の**発見** | どう直すべきかの**解法** |
| 言葉の中立性の点検（引っかかった語の指摘） | 掲載の可否・掲載基準の判断 |
| 設計仕様書への批判（テキストで手渡したもの） | 実装が正しいかの推測 |
| 変更差分への批判（テキストで手渡したもの） | データの正しさの検証 |
| 離脱しそうな箇所の指摘 | 動いているシステムの内部状態の推測 |

このページは**公開後の実物**（公開URL）を見せるときの手順です。
**マージ前の変更差分**を複数AIに点検させる手順は `tools/agents/REVIEW_CHARTER.md`（⑧査読班）
にあり、ここに書いた線引きをそのまま引き継いでいます。

### なぜこの線か

AIは**手渡した成果物の中に全部が写っているとき**は強い。レンダリング済みのページは
それに当たる（押しにくいボタンも読めない文字もすべて画面内にある）。一方、動いて
いるシステムの内部状態は写らないので、そこは推測になり、外す。

実例（2026-07-20）:
- 外した — 「stanceの重み付けが古い会期のまま」→ そんな構造は存在しなかった。
  「委員長報告のノイズ」→ 修正済み。「新党3党が未着手」→ 2党は組み込み済み。
  いずれも修正前のスクリーンショットから内部を推測したもの。3件中0件が有効。
- 当てた — 名簿仕様への4点批判。とくに「過去ログが統一会派の下にしかない新人は
  時間が経っても永久に自己修復しない」は正しく、こちらの想定が誤っていた。
  テキストで渡された仕様書を読んだもの。

### 解法を聞かない理由

モデル間で学習データが重なるため、**美意識まで相関する**。「どう直すべきか」を
複数に聞くと同じ方向に収束し（クリーム地＋セリフ見出し＋テラコッタの差し色、
角丸カード、絵文字の見出し記号）、合議による凡庸化が起きる。しかも一致するので
正しく見えてしまう。解法はサイトの目的から逆算して決める。

## 渡すもの

- 公開URL（実物。スクリーンショットだけだと見えない部分を推測で埋め始める）
- スマホ幅とPC幅の画面
- **明示的な禁止事項**（下記）

## テンプレート

```
以下のサイトをレビューしてください。
https://ai-seisaku-kurabe.github.io/

【このレビューで見ないでほしいこと】
- 実装・内部構造の推測
- データの正しさ（別の仕組みで機械的に検証しています）
- すでに修正済みかどうかの推測
画面から直接読み取れることだけを答えてください。

【質問】
1. トップに来て30秒で、このサイトが何をするものか分かりましたか。
   分からなかったなら、どこで分からなくなりましたか。
2. スマホで片手操作をしたとき、押しにくい・指が届かない箇所はどこですか。
3. 最後まで読まずに離脱するとしたら、どこですか。なぜですか。
4. 政治的に偏っている・誘導的だと感じた表現があれば、その語を原文のまま
   引用してください。（言葉の癖はモデルごとに違うので、複数モデルで
   突き合わせる価値が最も高い項目）
```

## 受け取ったあと

1. 複数モデルの指摘を突き合わせる。**1モデルだけが言ったこと**は保留、
   **複数が同じ箇所を指す**なら実在の可能性が高い。
2. 「内部状態の推測」に踏み込んだ指摘は、**採用前に必ずコードで実測する**。
   2026-07-20 の3件はすべてこれで棄却された。
3. 解法は自分で決める。指摘は「どこが問題か」までしか使わない。

## この規律の理由

このサイトの価値は「AIが一次情報を集め、判断はあなたに返します」にある。
AI同士の合意を根拠にすると、読者から検証できない層が増え、検証可能性が下がる。
AIの出力は、**機械的検証を通るか、人間が承認するか**のどちらかを経ずに公開しない。
AI同士の合意は承認の代わりにならない。
