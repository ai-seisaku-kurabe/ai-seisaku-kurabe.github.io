# 設計: GitHubアカウント不要のご意見フォーム

作成 2026-07-21 / 状態: 設計（未実装）

## 背景

現在の `feedback.html` は GitHub Issues に誘導している。ホスティングを Netlify から
GitHub Pages に移した際、Netlify Forms が使えなくなったための暫定措置で、ページ自身に
「GitHubのアカウントが必要です。お持ちでない方向けの入力フォームは準備中です。」と
書いてある。**有権者向けのサイトでありながら、意見を送るのに開発者向けサービスの
アカウントを要求している**状態を解消する。

## 決めたこと（ユーザー判断）

| 論点 | 決定 |
|---|---|
| 届いた意見の扱い | **非公開**。サイトには出さず、運営者だけが読む |
| 気づき方 | **新着件数だけ**を週次で通知（本文は通知経路に流さない） |
| 連絡先の受け取り | **受け取らない**。本文のみ。個別の返信はしない |
| 荒らし対策 | 連投制限 ＋ 緊急停止スイッチ ＋ 文字数上限（App Check は適用済み） |

## 方式の選定

**採用: Firestore に直接書き込む自作フォーム。**

既存の Firebase（Spark プラン・App Check 適用済み・サービスアカウント鍵登録済み）に
コレクションを1つ足すだけで、外部サービスを増やさずに済む。送信者のデータが第三者に
渡らない点が、一次情報を自前で持つというこのサイトの作りと整合する。

却下した案:
- **Google フォーム等の埋め込み** — 実装は最短だが、送信者のIP・ブラウザ情報が外部に
  渡り、Cookie 同意の問題も持ち込む。
- **GitHub Issues をアカウント不要にする** — 技術的に不可能。代理作成にはトークンが必要で、
  静的サイトではクライアントに置くしかない（＝公開＝悪用される）。中継サーバーを立てれば
  可能だが無料運用の前提が崩れる。

### 費用と継続性のリスク

- **課金は構造的に発生しない。** Spark プランは支払い方法が未登録で、無料枠を超えると
  請求ではなく**操作の拒否**になる。無料枠は書き込み 20,000件/日・保存 1 GiB。
- **見積もりは善意の利用者を前提にしてはならない。** 1送信＝1書き込みなので、通常の
  利用では枠に到達しないが、これは悪用時の見積もりになっていない。実際に検討すべきは
  次の2つ。
  - **書き込み枠（20,000件/日）を溶かされた場合**: その日は新規の意見が集計に
    入らなくなる。ただし読み取りは別枠なので、「みんなの結果」の**表示は続く**。
    書き込み枠は**翌日リセットされる**。
  - **保存容量（1 GiB）を溶かされた場合が本命のリスク。** 日本語2000字は約6 KB。
    書き込み枠の上限いっぱい（20,000件/日）で送り続けられると 20,000 × 6 KB ＝
    約120 MB/日で、**約8日の継続攻撃で 1 GiB に到達する**。保存容量は
    **書き込み枠と違って日次リセットされない**。到達すると `feedback` だけでなく
    `aggregates/summary` への書き込みも含めて**恒久的に拒否**され、復旧には
    Firebase コンソールから数万件のドキュメントを手で削除する必要がある。
- **無料枠は Firestore の集計機能と共有**している。上記いずれの経路でも、荒らしで
  枠を溶かされると「みんなの結果」の更新（保存容量の場合は表示も）まで巻き添えになる。
  これが実務上の最大リスクで、停止スイッチを入れる理由になっている。
  ただし停止スイッチの実効性には限界がある（後述「停止スイッチの限界」）。
- **プラン内容が変わる可能性は実在する。** 2026年2月3日に Cloud Storage が Spark から
  外され Blaze 必須になった前例がある（このサイトは Cloud Storage を使っていないため
  影響を受けていない）。将来 Firestore が同様になった場合でも、**サイト本体は
  GitHub Pages なので無傷**で、止まるのは送信と集計だけ。政策データの閲覧には波及しない。
- 保存データは小さく、いつでも吸い出して別手段に移せる。

## Firestore のルール（この設計で最も重要な箇所）

```
match /feedback/{id} {
  allow create: if request.resource.data.keys().hasOnly(['text','from','at'])
             && request.resource.data.text is string
             && request.resource.data.text.size() > 0
             && request.resource.data.text.size() <= 2000
             && request.resource.data.from is string
             && request.resource.data.from.size() <= 40
             && request.resource.data.at == request.time;
  allow read, update, delete: if false;
}
```

`read: if false` が要。**サイトから読めないだけでなく、クライアント経路のどこからも
読めない**。運営者が読む経路は Firebase コンソールと、ルールを迂回できるサービス
アカウント（件数集計）の2つだけになる。

`hasOnly` で項目を限定するのは、任意のフィールドを詰めた巨大ドキュメントを書き込まれる
のを防ぐため。`at == request.time` はサーバー時刻の強制で、クライアントが時刻を
詐称できないようにする。

既存の `aggregates/summary` のルールには「※テスト用。悪用対策は App Check の追加を推奨」
というコメントが残っているが、App Check は 2026-07-20 に適用済みなので、この機会に
コメントを実態に合わせる。

## 保存する内容

```
{ text: "本文（2000字まで）", from: "送信元ページ名（例: guide）", at: サーバー時刻 }
```

**IP・User-Agent・メールアドレス・氏名は保存しない。** `from` はページ名だけで、
個人の特定に使えない。これにより「個人情報は保存しない」という公開中の宣言を
書き換えずに維持できる。ドキュメントIDは自動生成（ランダム）。

## 画面の動き

- **送信前** — 「返信はできません」「個人情報は書かないでください」「いただいた内容は
  公開しません」を明示する。
- **送信中** — ボタンを無効化（二重送信防止。既存の実装と同じ作り）。
- **成功** — お礼を表示してフォームを消す。
- **失敗** — エラー文言と併せて **GitHub Issues への導線**を出す。フォームが使えなく
  なっても受付がゼロにならないための二重化で、Issues の導線は平常時も残す。
- **連投制限** — `localStorage` に最終送信時刻を持ち、60秒以内は送信させない。
  突破は可能だがハードルにはなる。App Check が機械的な連投を、これが人手の連投を抑える。
- **停止中** — `config.json` の `feedback_enabled` が false なら、フォームを出さずに
  理由だけ表示する（Issues 導線は残す）。**再ビルド不要**で、既存の選挙期間スイッチと
  同じ「実行時に config.json を fetch する」方式に合わせる。

### 停止スイッチの限界

`config.json` の `feedback_enabled` は**クライアント側の分岐にすぎない**。
`firebase.js` の設定値（apiKey / projectId など）は公開HTMLに同梱されているため、
攻撃者は `feedback.html` を経由せず、Firestore の SDK を直接呼び出して `feedback`
コレクションに書き込める。この経路では `config.json` を見ないので、**`feedback_enabled`
を false にしても Firestore への書き込みは1件も止まらない**。

- **クライアント側スイッチ（`config.json`）の役割は「利用者への告知」であって
  「防御」ではない。** 「いま受付を止めています」と正直に伝える機能であり、
  悪意ある書き込みを防ぐ機能ではない。
- **緊急時に書き込みを実際に止める唯一の手段は、Firebase コンソールで Firestore の
  ルールを書き換えることだけ。** `feedback` の `allow create` を `if false` に
  変更して公開する（「Firestore のルール」節・Task 1 参照）。これはコンソール上の
  手作業で、リポジトリへのコミットでは反映されない。
- **`config.json` 経由の停止は fail-open（安全側に倒れない）設計でもある。**
  `fetch('config.json')` が失敗すると `cfg` は `null` のままになり、`stopped()` は
  false を返す ＝ **フォームは有効化される**。オフライン・キャッシュ・GitHub Pages
  側の障害など fetch が失敗する状況では、止めたつもりのフォームが動き続ける。
  これは可用性を優先した選択として妥当だが、**「確実に止まる」保証ではない**ことを
  明記しておく。同じ理由で、GitHub Pages のキャッシュや、変更前から開かれている
  タブにも `config.json` の更新は即座には効かない。
- 上記のいずれも、**Firestore のルール変更だけが確実な停止手段**であるという
  結論を変えない。運用手順は「実装後に残る手作業」に記載する。

## 新着件数の通知

`tools/agents/feedback_count.py` が、`snapshot_archive.py` と同じサービスアカウント認証
（`FIREBASE_SA_KEY`）で `feedback` コレクションの件数を取得する。

**本文は取得しない。** GitHub Actions のログや成果物に意見の中身が残ると、非公開にした
意味がなくなるため、取得するのは件数のみとする。

数えるのは**総件数ではなく「前回確認した時刻より後に届いた件数」**とする。
`tools/state/feedback_state.json` に保存するのは件数ではなく**最終確認時刻**で、
集計クエリに `at > 最終確認時刻` の条件を付ける。

総件数の差分で新着を出す方式は**採らない**。読んだ意見をコンソールから削除すると
総件数が減り、差分がマイナスになって新着を検知できなくなるため（10件→3件読んで削除→
7件、で差分 −3）。**読んだら消す運用と、総件数の差分方式は両立しない。**
時刻で絞れば削除の影響を受けず、本文を取得しない条件も維持できる。

結果は「新着N件」としてコミットメッセージと実行サマリに残す。運営者はコミット履歴で
気づける。

`.github/workflows/feedback-count.yml` は週次 cron ＋ `workflow_dispatch`。

## 開示（about.html）

「ご意見の扱い」を1項目追加する。内容は、非公開であること・個別の返信はできないこと・
個人情報を保存しないこと・GitHub Issues なら公開の場で議論できること。

**これは方法論への追記＝憲法にあたる変更**なので、`EDITOR.md` の規定に従い、
直接プッシュせず PR を出して人の承認を待つ。

## 検証

1. **ブラウザから `feedback` コレクションを読もうとして拒否されること**（最重要。
   ルールの取り違えは、寄せられた意見が全世界から読める事故に直結する）。
2. 送信 → Firebase コンソールに1件届く → 削除できる。
3. `config.json` を停止に切り替えると、再ビルドせずに受付が消える。
4. 60秒以内の連投が止まる。2001文字がルール側で拒否される。
5. `feedback_count.py` が件数を返し、**本文がログに出ていない**こと。
   **削除しても壊れないこと** — 3件送る→1件をコンソールで削除→再実行して、
   新着の数え方が総件数の減少に引きずられないことを確認する。
6. `health_check.py` に「`feedback.html` にフォームが存在するか」の監視を追加する
   （生成の取りこぼしで、いつの間にか受付が消えている事故を防ぐ）。

## 実装後に残る手作業

- **意見を読む**のは Firebase コンソール（`feedback` コレクション）。読んだら削除する。
- **荒らされたとき（緊急停止）** — `config.json` の `feedback_enabled` を `false` に
  してコミットするのは、あくまで**利用者への告知**（「停止スイッチの限界」参照）。
  これだけでは Firestore への書き込みは止まらない。**書き込みを実際に止めるには、
  Firebase コンソール → Firestore Database → ルール で `feedback` の
  `allow create` を `if false` に書き換えて公開する。** 反映まで最大1分ほど。
  手順:
  1. Firebase コンソールでルールを `allow create: if false;` に書き換えて公開する
     （書き込みを止める。これが唯一の確実な停止手段）。
  2. `config.json` の `feedback_enabled` を `false` にしてコミットする
     （利用者に「止めていること」を伝える。再ビルド不要。ただし fail-open のため
     即座には全員に効かない）。
  3. 荒らしが収まったら、ルールを元の内容（本設計の「Firestore のルール」節）に
     戻し、`config.json` を `true` に戻す。

## この設計で「やらないこと」

- 個別の返信、メール通知、意見の公開、添付ファイルの受け取り。
- 意見に対するAIの自動応答・自動分類。⑦応答班は将来の枠として予約されているが、
  この設計には含めない。
